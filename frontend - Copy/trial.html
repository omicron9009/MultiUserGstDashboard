Foundation · 01

What changes at industry grade — and why

Your previous schema was solid for getting started. Industry grade means it can survive audits, developer changes, GST API responses changing shape, and your team growing from 1 to 10 people — without ever losing data or breaking existing queries.

The single biggest shift: Stop thinking about tables as "storage". Start thinking about them as immutable event logs. In GST compliance, every API call is an event. That event happened at a specific time. You never want to lose it — not even if you fetch "fresher" data the next day. Financial regulators, CA firms, and your own debugging will thank you for this.
Design concern Basic approach Industry-grade approach Why it matters
Storing data UPSERT — overwrite old with new ✦ Append-only log + latest-view If GST portal shows stale data and you re-fetched, you've lost the original. Disputes, audits, reconciliation need history.
Schema changes ALTER TABLE, hope nothing breaks ✦ Versioned migrations (Alembic/Flyway) Every DB change is tracked, reversible, and deployed consistently across dev/staging/prod.
Table ownership One DB, tables named by service ✦ One PostgreSQL schema per service Same DB cluster, total logical isolation. gstr1.b2b vs gstr3b.details. Cross-service queries still possible in SQL.
Raw data JSONB column on main table ✦ Dedicated raw_events table, never touched after insert Separation of concerns. Parsed data can be re-generated from raw. Raw is your source of truth.
Idempotency Assume API won't be called twice ✦ request_id / idempotency_key column Network retries, background jobs, and human mistakes will cause duplicate calls. Your DB must handle this without exploding.
Error tracking Return dict with success=False ✦ api_calls log table — every call, success or fail You can replay failed fetches, audit API usage, and diagnose intermittent GST API errors.
Computed totals Parse JSONB at query time ✦ Materialized view or generated columns Fast reporting queries on SUM(igst_total) without unpacking JSON every time.

Core Pattern · 03

Audit trail — immutability as a first-class citizen

GST compliance is a legal domain. Data you fetched on March 1st might be different from the same endpoint on March 15th (amendments, revisions). You must be able to show: "On this date, this is what the GST portal returned."

SQL gstr1/raw_events — one insert per API call, zero updates ever
CREATE TABLE gstr1.raw_events ( -- Identity id BIGSERIAL PRIMARY KEY, idempotency_key TEXT UNIQUE NOT NULL, -- prevents duplicate inserts -- What was fetched gstin TEXT NOT NULL, endpoint TEXT NOT NULL, -- 'b2b', 'b2cs', 'cdnr'... year TEXT NOT NULL, month TEXT NOT NULL, -- The API response — frozen forever http_status INTEGER, gst_status_cd TEXT, -- '0'=error, '1'=success payload JSONB NOT NULL, -- When & how fetched_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), fetch_duration_ms INTEGER, -- API latency tracking triggered_by TEXT -- 'auto_refresh', 'manual', 'api_request' ); -- Fast lookups: "all b2b fetches for this GSTIN in March" CREATE INDEX ON gstr1.raw_events (gstin, endpoint, year, month, fetched_at DESC);
The idempotency_key is critical. Build it as: f"{gstin}:{endpoint}:{year}:{month}:{timestamp_rounded_to_minute}". If your background job retries due to a network blip, the second INSERT will silently fail (ON CONFLICT DO NOTHING) — no duplicates, no errors, no data corruption.
Core Pattern · 04

Schema-per-service — one PostgreSQL, total isolation

Instead of separate databases (7 connection strings, 7 backup configs, 7 permission setups), use one PostgreSQL cluster with separate schemas. This is the industry standard used by every serious fintech at scale.

❌ Multiple databases
7 connection strings. No SQL joins across services. Complex backups. Hard to manage permissions. Overkill for this problem size.
✅ One DB, multiple schemas
1 connection string. SQL joins across services when needed. One backup. One permission model. Isolated ownership per service.
SQL Schema layout — one CREATE SCHEMA per service
-- One PostgreSQL database: gst_platform -- One schema per service. Total logical isolation. CREATE SCHEMA core; -- sessions, taxpayers, api_calls log CREATE SCHEMA gstr1; -- all GSTR-1 tables CREATE SCHEMA gstr2a; -- all GSTR-2A tables CREATE SCHEMA gstr2b; -- all GSTR-2B tables CREATE SCHEMA gstr3b; -- all GSTR-3B tables CREATE SCHEMA gstr9; -- all GSTR-9 tables CREATE SCHEMA ledger; -- all ledger tables CREATE SCHEMA analytics;-- materialized views only -- Tables are addressed as: schema.table -- gstr1.b2b → GSTR-1 B2B state table -- gstr1.raw_events → GSTR-1 raw API log -- gstr3b.details → 3B state table -- Cross-service query (still pure SQL!) SELECT g1.total_igst AS gstr1_igst, g3.igst_payable AS gstr3b_igst FROM gstr1.b2b g1 JOIN gstr3b.details g3 ON g1.gstin = g3.gstin AND g1.year = g3.year AND g1.month = g3.month WHERE g1.gstin = '27AABCU9603R1ZX';
Core Pattern · 05

Event sourcing lite — the fetch log

Every time any service function is called, one row goes into core.api_calls. This is your system's heartbeat. It tells you: what was fetched, when, by whom, how fast, and whether it succeeded.

SQL core.api_calls — the system's audit heartbeat
CREATE TABLE core.api_calls ( id BIGSERIAL PRIMARY KEY, -- Who / what / when gstin TEXT NOT NULL, service TEXT NOT NULL, -- 'gstr1', 'gstr3b', 'ledger' function_name TEXT NOT NULL, -- 'get_gstr1_b2b', 'get_gstr3b_details' year TEXT, month TEXT, -- Result success BOOLEAN NOT NULL, http_status INTEGER, gst_status_cd TEXT, error_code TEXT, error_message TEXT, -- Performance duration_ms INTEGER, called_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), triggered_by TEXT -- 'scheduler', 'webhook', 'manual' ); -- Useful queries this enables: -- "Show me all failed gstr1 fetches for this GSTIN this week" -- "What's the average GST API response time?" -- "Which GSTINs haven't been refreshed in 6 hours?" CREATE INDEX ON core.api_calls (gstin, called_at DESC); CREATE INDEX ON core.api_calls (success, called_at DESC) WHERE NOT success;
Core Pattern · 06

Idempotency — calling the same function twice is safe

Your session_refresh_manager runs on a schedule. Users can trigger manual fetches. The GST API can be slow and cause retries. Without idempotency, you get duplicate rows. With it, the second call is a no-op.

SQL + Python Idempotent upsert pattern for state tables
-- State table (Layer 2): one row per (gstin, year, month), always current INSERT INTO gstr1.b2b ( gstin, year, month, records, invoice_count, total_taxable, total_igst, total_cgst, total_sgst, last_raw_event_id, fetched_at ) VALUES ( %(gstin)s, %(year)s, %(month)s, %(records)s, %(count)s, %(taxable)s, %(igst)s, %(cgst)s, %(sgst)s, %(raw_event_id)s, NOW() ) ON CONFLICT (gstin, year, month) DO UPDATE SET records = EXCLUDED.records, invoice_count = EXCLUDED.invoice_count, total_taxable = EXCLUDED.total_taxable, total_igst = EXCLUDED.total_igst, total_cgst = EXCLUDED.total_cgst, total_sgst = EXCLUDED.total_sgst, last_raw_event_id = EXCLUDED.last_raw_event_id, fetched_at = NOW(); -- Raw events table (Layer 1): idempotency_key prevents duplicate rows INSERT INTO gstr1.raw_events (idempotency_key, gstin, endpoint, year, month, payload, ...) VALUES (%(key)s, ...) ON CONFLICT (idempotency_key) DO NOTHING; -- silent no-op if already exists
Implementation · 07

The universal table template

Every single service function gets two tables following this exact template. Copy-paste this and fill in the function-specific columns. Consistency is the point — any developer can read any table and immediately understand it.

SQL Template — replace SCHEMA and TABLE_NAME for each function
-- ════════════════════════════════════════════════════════════════ -- LAYER 1: Raw event log (append-only, immutable) -- Naming: {schema}.raw_events (shared per schema — all endpoints) -- ════════════════════════════════════════════════════════════════ CREATE TABLE IF NOT EXISTS {SCHEMA}.raw_events ( id BIGSERIAL PRIMARY KEY, idempotency_key TEXT UNIQUE NOT NULL, gstin TEXT NOT NULL, endpoint TEXT NOT NULL, year TEXT, month TEXT, http_status INTEGER, gst_status_cd TEXT, payload JSONB NOT NULL, fetched_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), duration_ms INTEGER, triggered_by TEXT ); CREATE INDEX IF NOT EXISTS ON {SCHEMA}.raw_events (gstin, endpoint, year, month, fetched_at DESC); -- ════════════════════════════════════════════════════════════════ -- LAYER 2: Parsed state table (one row per identity key, upserted) -- Naming: {schema}.{endpoint} e.g. gstr1.b2b, gstr3b.details -- ════════════════════════════════════════════════════════════════ CREATE TABLE IF NOT EXISTS {SCHEMA}.{TABLE_NAME} ( -- Identity (Primary Key — never changes) gstin TEXT NOT NULL, year TEXT NOT NULL, month TEXT NOT NULL, PRIMARY KEY (gstin, year, month), -- ↓↓ ADD FUNCTION-SPECIFIC COLUMNS HERE ↓↓ {specific_column_1} NUMERIC, -- e.g. total_igst {specific_column_2} INTEGER, -- e.g. invoice_count records JSONB, -- structured parsed data array -- ↑↑ END FUNCTION-SPECIFIC COLUMNS ↑↑ -- Lineage (links back to source of truth) last_raw_event_id BIGINT REFERENCES {SCHEMA}.raw_events(id), -- Timestamps fetched_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), created_at TIMESTAMPTZ NOT NULL DEFAULT NOW() ); CREATE INDEX IF NOT EXISTS ON {SCHEMA}.{TABLE_NAME} (gstin, fetched_at DESC);
Note last_raw_event_id: This foreign key links every state row back to the exact raw API response that produced it. If you want to re-parse or audit a specific fetch, you can do: SELECT payload FROM gstr1.raw_events WHERE id = (SELECT last_raw_event_id FROM gstr1.b2b WHERE gstin=... AND year=... AND month=...). Perfect traceability.
Implementation · 08

Migration strategy — Alembic

Never run raw SQL ALTER TABLE in production. Every schema change is a migration file — version-controlled, reversible, and applied consistently. This is how every serious Python project manages database changes.

📁

File-based versioning

Each change is a .py file: 0001_create_core_schema.py, 0012_add_gstr1_b2b.py. Committed to Git. Every developer applies the same changes.

Alembic
↩️

Reversible changes

Every migration has an upgrade() and downgrade() function. If a deployment goes wrong, you roll back with one command.

upgrade / downgrade
➕

Adding a new function

Run alembic revision --autogenerate -m "add_gstr1_txp". It detects the new table in your models and writes the migration automatically.

autogenerate
🔒

Non-breaking changes only

New columns: always DEFAULT NULL or with a default. Drop old columns: never immediately — deprecate first. Old code keeps working.

backwards compatible
Python Alembic migration example — adding gstr1_txp table
# migrations/versions/0015_add_gstr1_txp.py # This file is auto-generated, then you review & commit it to Git. def upgrade(): op.execute("CREATE SCHEMA IF NOT EXISTS gstr1") op.create_table( 'txp', sa.Column('gstin', sa.Text(), nullable=False), sa.Column('year', sa.Text(), nullable=False), sa.Column('month', sa.Text(), nullable=False), sa.Column('records', JSONB()), sa.Column('total_taxable', sa.Numeric()), sa.Column('last_raw_event_id', sa.BigInteger(), nullable=True), sa.Column('fetched_at', TIMESTAMP(timezone=True), server_default=func.now()), sa.Column('created_at', TIMESTAMP(timezone=True), server_default=func.now()), sa.PrimaryKeyConstraint('gstin', 'year', 'month'), schema='gstr1' ) def downgrade(): op.drop_table('txp', schema='gstr1')
Implementation · 09

What NOT to do — anti-patterns to avoid

  • 🚫
    Don't store money as FLOAT. Use NUMERIC(18, 2) for all tax amounts. Floating point has rounding errors — ₹0.01 discrepancies in a CA firm's audit report will cause chaos.
  • 🚫
    Don't delete raw events. Ever. The raw table is append-only. If you fetched bad data, the next fetch overwrites the state table. But the bad raw response stays in history — it's evidence of what the GST portal returned.
  • 🚫
    Don't put business logic in SQL triggers. Use the Python service layer for transformations. SQL triggers are invisible, hard to test, and make debugging a nightmare when the GST API changes its response format.
  • 🚫
    Don't use TEXT for year/month if you ever need date math. Either store as TEXT consistently (simple, safe) OR store as a proper DATE (first day of month). Never mix — inconsistency here causes silent bugs in period comparisons.
  • 🚫
    Don't put all raw_events in one global table. Keep them schema-scoped (gstr1.raw_events, gstr3b.raw_events). A single global table becomes a 100M-row monster and loses the isolation benefit entirely.
  • 🚫
    Don't skip the triggered_by column. When something goes wrong at 3am, you need to know whether the bad data came from your scheduler, a user's manual request, or a webhook. Without this, debugging takes 10x longer.
  • 🚫
    Don't store access tokens without an expiry column. Your core.sessions table must have token_expires_at. Without it, the session_refresh_manager has to re-decode every JWT on every refresh cycle instead of just querying expiring sessions.

Production · 11

Python layer — the repository pattern

In the service layer, separate the "fetch from GST API" logic from the "save to DB" logic. The repository pattern makes this clean, testable, and consistent across all services.

Python repositories/gstr1_repo.py — DB layer, zero business logic
from datetime import datetime, timezone import hashlib, json class Gstr1Repository: """All DB operations for gstr1 schema. No business logic here.""" def __init__(self, conn): self.conn = conn def _idempotency_key(self, gstin, endpoint, year, month, fetched_at): # Stable key: same gstin+endpoint+period in same minute = same key raw = f"{gstin}:{endpoint}:{year}:{month}:{fetched_at.strftime('%Y%m%d%H%M')}" return hashlib.sha256(raw.encode()).hexdigest()[:32] def insert_raw_event(self, gstin, endpoint, year, month, payload, http_status, gst_status_cd, duration_ms, triggered_by): """Layer 1: append-only insert. Returns the new row id.""" now = datetime.now(timezone.utc) key = self._idempotency_key(gstin, endpoint, year, month, now) with self.conn.cursor() as cur: cur.execute(""" INSERT INTO gstr1.raw_events (idempotency_key, gstin, endpoint, year, month, http_status, gst_status_cd, payload, duration_ms, triggered_by) VALUES (%s,%s,%s,%s,%s,%s,%s,%s,%s,%s) ON CONFLICT (idempotency_key) DO NOTHING RETURNING id """, (key, gstin, endpoint, year, month, http_status, gst_status_cd, json.dumps(payload), duration_ms, triggered_by)) row = cur.fetchone() return row[0] if row else None def upsert_b2b(self, gstin, year, month, parsed, raw_event_id): """Layer 2: upsert current best-known state.""" with self.conn.cursor() as cur: cur.execute(""" INSERT INTO gstr1.b2b (gstin, year, month, invoice_count, total_taxable, total_igst, total_cgst, total_sgst, total_cess, records, last_raw_event_id) VALUES (%s,%s,%s,%s,%s,%s,%s,%s,%s,%s,%s) ON CONFLICT (gstin, year, month) DO UPDATE SET invoice_count = EXCLUDED.invoice_count, total_taxable = EXCLUDED.total_taxable, total_igst = EXCLUDED.total_igst, total_cgst = EXCLUDED.total_cgst, total_sgst = EXCLUDED.total_sgst, total_cess = EXCLUDED.total_cess, records = EXCLUDED.records, last_raw_event_id = EXCLUDED.last_raw_event_id, fetched_at = NOW() """, (gstin, year, month, parsed['invoice_count'], parsed['total_taxable'], parsed['total_igst'], parsed['total_cgst'], parsed['total_sgst'], parsed['total_cess'], json.dumps(parsed['records']), raw_event_id)) self.conn.commit()
